<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local repo → npm registry | Publish is irreversible (versions cannot be unpublished after 72h); a bad publish poisons the registry |
| local repo → GitHub release | Tag pushes are mostly irreversible (can be force-deleted but URL persists in caches/forks) |
| npm dist-tag mechanism → user installs | The `--tag rc` flag is the gate; missing it breaks the entire feedback+cooling-off design |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Missing `--tag rc` flag → bad latest dist-tag | mitigate | Acceptance criterion verifies `npm view donnyclaude@latest version` does NOT return 1.2.0-rc.1; if it does, the publish is broken and Donovan deprecates the bad version before proceeding |
| T-04-02 | Information Disclosure | npm token leakage in CI logs | accept | This plan runs interactively on Donovan's machine, not CI; npm credentials are stored locally; standard npm auth flow |
| T-04-03 | Spoofing | Wrong commit tagged | mitigate | Tag target is `HEAD` (the version-bump commit on top of the prune); pre-tag verification confirms `HEAD~1` is the prune commit; tag verification post-publish checks `git log v1.2.0-rc.1 -1` |
| T-04-04 | Repudiation | Release notes change after publish | accept | GitHub release notes can be edited but the original is in git history (the release notes themselves don't get committed to the repo, but the audit trail is in PRUNE-LOG.md which IS committed) |
| T-04-05 | Tampering | npm tarball includes packages/_archived-skills/ which inflates size | accept | Per D-23, archived skills are co-located with packages/skills/ and shipped in the tarball. The install path skips them. Tarball size is acceptable per locked decision |
</threat_model>
